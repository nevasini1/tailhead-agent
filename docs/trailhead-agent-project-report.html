trailhead-agent

Disclaimer: Engineering / product narrative. Not peer-reviewed research. Deep sections synthesize design docs and implementation intent.

1. Executive summary

What it is: A Python CLI that combines (A) Playwright discovery of Trailhead /content/learn/ unit links under strict URL policy, (B) LLM ranking with Pydantic-validated JSON and href allowlisting (no invented URLs), orchestrated by a mandatory LangGraph graph with optional planner and repair passes, and (C) optional org helpers (YAML checklists, sf visibility via CliOrgExecutor, org prepare --open-playground for browser + human Launch Playground).

Explicit non-goals: Auto-submit quizzes, credential stuffing, automatic Launch Playground clicks, crawling outside allowed Trailhead paths.

  • Observability: trace_id, duration_ms, agent_version; --log-json; E2E JSON + .webm + e2e-report.html.
  • Demos: Title-card phases (inputs, discovery, LLM order slide, next steps, walk); stderr narration (TRAILHEAD_DEMO_NARRATION).

2. Goals and problem fit

  • Assist learners with navigation and prioritization using natural-language intent.
  • Do not automate quizzes, knowledge checks, or credential abuse.
  • Separate concerns: browser discovery vs. LLM reasoning vs. future Salesforce org automation.

3. Stakeholders and value

Learners        Intent-ordered queue (e.g. hands-on-first) vs. scanning long TOCs manually
Demos / L&D     Reproducible plan-primary vs org-prepare clips + JSON for narrative
Operators       Bounded URLs, exit codes, headless CI, JSON logs for pipelines
Extensibility   OrgExecutor protocol: noop today, cli/sf hooks without rewriting ranker

4.5 End-to-end data flow

start_url
  -> Playwright: collect_units_for_seed -> List[UnitRef] (title, href; reason empty from scrape)
  -> LangGraph.invoke: intent + candidates as dict rows
  -> Optional planner notes merged into payload (LLM_PLANNER_PHASE=1)
  -> Primary rank LLM call -> parse -> Pydantic validate -> materialize ordered hrefs subset
  -> If invalid / empty allowlisted rows and repair enabled: repair LLM with excerpt + hints
  -> finalize: emit result_units / result_hrefs; runner maps back to UnitRef with reasons
  -> RunPlan JSON (+ optional walk_ranked navigations for recording)

5. LangGraph ranking

5.1 Nodes

prepare     Builds user JSON: intent, candidates[], ranking_user_message_context (execution_steps, hard_rules)
planner     If enabled: extra LLM notes only; skipped when LLM_PLANNER_PHASE != 1
rank_primary Single structured completion; ordered_units must use discovered hrefs only
rank_repair If needs_repair and LLM_RANKING_REPAIR: second completion with invalid_output excerpt
finalize    Choose repair vs primary output; fallback to discovery order if nothing valid

5.2 Why a graph vs one shot

  • Repair path is conditional: avoids paying for repair on happy path while recovering from schema drift or model mistakes.
  • Planner is optional: adds latency/cost only when operators enable it.
  • Single orchestrator (mandatory LangGraph) keeps ranking behavior consistent across providers.

6. Discovery subsystem

  • Lazy loading: TRAILHEAD_DISCOVERY_SCROLL_ROUNDS merges anchors that appear after scroll.
  • Pagination: TRAILHEAD_DISCOVERY_PAGINATION_CLICKS for Load more / Show more-style controls (bounded).
  • Navigation: TRAILHEAD_DISCOVERY_NAV_WAIT_UNTIL (e.g. domcontentloaded vs load).
  • Failure semantics: Zero usable units or degenerate TOC → DiscoveryError with hints; no LLM call (saves cost and avoids ranking garbage).
  • Blockers: Login walls, geo, error pages can be flagged for operator action.

7. Org and Playground

  • plan produces ranked JSON + optional browser walk on Trailhead only.
  • org prepare --open-playground opens the module URL, title cards, dwell/scroll; does not run LangGraph ranking.
  • Launch Playground remains a human action; TRAILHEAD_RECORDING_PLAYGROUND_CTA_SCROLL only scrolls CTAs into view for clearer recordings.
  • playground_ready in JSON reflects executor/sf checks, not “browser clicked Launch.”
  • Checklists map unit context to YAML steps (config/org_checklists.yaml) for hands-on guidance offline from the ranker.

11. Impact analysis

11.1 Intended positive impact

  • Learner throughput: Converts “read entire TOC” into a single ranked list aligned to stated intent.
  • Interop: JSON plan is stable input for checklists, internal LMS hooks, or human review.
  • Trust in demos: Narration + title cards + manifest reduce “black box” perception of the agent.
  • Boundary clarity: Explicit non-goals reduce misuse and support compliance conversations.

11.2 Limits and negative externalities

  • Vendor coupling: Ranking quality and availability depend on LLM APIs and pricing.
  • Maintenance: Trailhead UI changes are an ongoing tax on discovery, not a one-time parse.
  • No certified pedagogy: Order is model-opinionated; not curriculum-authoritative without human review.

12. Novelties and differentiators

  • Hard grounding: Output hrefs ⊆ discovered hrefs — strong constraint vs free-form “suggested links.”
  • Graph-shaped LLM workflow: Conditional repair and optional planner as first-class control flow.
  • Discovery as typed errors: DiscoveryError with structured details and hints for operators.
  • Demo ergonomics: Plan vs org visual language (cyan vs purple), epilogue navigation, friendly video filenames.
  • Org separation: Ranking pipeline never implied to “complete labs”; prepare is explicitly adjacent.

13. Research positioning and evaluation gaps

Not claimed: New neural architectures, SOTA accuracy on a public Trailhead benchmark (none shipped in-repo).

Applied research adjacency: Constrained decoding / structured output; retrieval-grounded generation (candidates as retrieval set); policy-bound web agents; educational integrity.

13.1 What a rigorous evaluation would add

Gold set        Human expert ordering for N (start_url, intent) pairs frozen at a DOM snapshot
Metrics         Kendall tau / pairwise win rate vs gold; nDCG if partial relevance labels
Ablations       Planner on/off; repair on/off; model family swap; temperature
Cost curve      Tokens per plan; p95 latency (discovery vs LLM vs total)
Robustness      DOM snapshot replay vs live site drift (monthly re-run)

14. Ethics and misuse resistance

  • Scope documentation and CLI boundaries reduce “take my quiz” expectations.
  • Allowlisted hrefs limit exfiltration of arbitrary attacker-chosen URLs via model output (still does not eliminate all prompt risks).
  • Human-in-the-loop org steps preserve accountability for org actions.

15. Roadmap and tradeoffs

15.1 Roadmap

  1. Richer OrgExecutor (deploy, Tooling API, guided org browser).
  2. Optional second-pass LLM checklist extraction from unit HTML (still HITL).
  3. Disk cache keyed by (start_url, intent_hash).
  4. Optional Pydantic Settings for env.

15.2 Tradeoffs

  • LLM vs rules: Better intent flexibility vs token cost and uptime dependency.
  • Headless vs headed: CI vs SSO realism.
  • Scope: Full Trailhead+org automation would imply legal, ToS, and selector SLAs beyond this codebase.

18. Conclusion

trailhead-agent is a bounded assistant: strong URL and output constraints, observable runs, and a deliberate split between “what to study next” (plan) and “how to open Trailhead / org” (prepare). Its deepest contributions are systems integration, safety framing, and operator-grade artifacts, not new core ML. Formal evidence would require gold rankings and measured ablations outside the current repository scope.
